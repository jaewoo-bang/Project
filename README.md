# 🎬🎵 CineBeat — AI 기반 음악 & 영화 개인화 추천 서비스

> **프롬프트 엔지니어링 실습 발표**
> 개발 기간: 2026.07.03 (당일 개발 · Vibe Coding)
> 개발 도구: Antigravity (AI Coding Assistant) + Supabase + Vanilla JS

---

## 1. 무엇을 만들려 했는가

### 프로젝트 목표

사용자가 **좋아하는 장르를 선택**하면, 그에 맞는 영화와 음악을 추천해주는
**개인화 콘텐츠 추천 서비스**를 만들고자 했습니다.

단순한 목록 나열이 아닌, 실제 서비스처럼 **회원가입 → 장르 선택 → 맞춤 추천**이
자연스럽게 이어지는 사용자 흐름을 갖추는 것이 핵심 목표였습니다.

---

### 구현한 기능들

| 기능 | 설명 |
|------|------|
| **회원가입 / 로그인** | 이메일 + 비밀번호 기반 인증 (Supabase Auth) |
| **장르 선호도 선택** | 가입 시 Action, Romance, Comedy 등 영화/음악 장르 다중 선택 |
| **개인화 추천 홈** | 선택한 장르 기반으로 영화 & 음악 카드 추천 |
| **카드 이미지 자동 연동** | 장르 선택 시마다 해당 콘텐츠의 실제 포스터/앉범 카버 표시 |
| **둘러보기 탭** | 전체 영화 30종 + 음악 30종을 페이지네이션 + 검색/필터로 탐색 |
| **상세 미리보기 모달** | 카드 클릭 시 YouTube 트레일러 / MV 인라인 재생 |
| **마이페이지** | 가입 정보 확인 + 장르 수정 + 회원탈퇴 |
| **아이디 / 비밀번호 찾기** | 전화번호 기반 계정 찾기 기능 |

---

### 기술 스택

```
Frontend   : HTML5 · Vanilla CSS · Vanilla JavaScript
Backend/DB : Supabase (PostgreSQL + Auth + Realtime)
이미지 소스 : IMP Awards · Daum 영화 CDN · Deezer CDN (공식 앉범 칋버)
서버        : npx serve (로컬 정적 서버)
버전관리    : GitHub
```

## 2. 프롬프트는 어떤 방식으로 작성했는가

### 핵심 전략: 세 가지 레이어

```
① 안드레이 카파시(Andrej Karpathy) 접근법
        ↓
② LLM에게 프롬프트를 잘 작성하게 하는 메타 프롬프트
        ↓
③ 프롬프트 블루프린터 (Blueprint) — 기초 작업
```

---

### ① 안드레이 카파시의 접근법

카파시는 LLM을 사용할 때 "소프트웨어 1.0 방식으로 명령하지 말고,
LLM이 스스로 생각하고 판단하도록 문맥을 충분히 주라"고 강조합니다.

> "Don't tell the AI what to do step by step.
> Tell it the goal, the constraints, and let it figure it out."
> — Andrej Karpathy

이 원칙에 따라, 저는 구체적인 코드 명령 대신
**"무엇을 만들고 싶은지 + 왜 필요한지 + 어떤 결과를 원하는지"**
를 중심으로 프롬프트를 작성했습니다.

**예시:**
```
❌ Bad:  "app.js의 renderCard 함수 214번째 줄에 webImage 속성을 추가해줘"

✅ Good: "회원가입할 때 장르마다 다양한 음악과 영화들이 나올 건데
          회원가입할 때마다 카드 이미지가 바꾵야 하고
          마이페이지에서 장르 수정하는 경우에도 이미지가 바꾵야 하니까
          이거에 맞게 해결해줘"
```

---

### ② LLM에게 프롬프트를 잘 작성하게 하는 메타 프롬프트

개발 시작 전, AI에게 직접 **"내 프로젝트를 위한 최적의 프롬프트 전략을 설계해줘"** 라고 요청했습니다.

이를 통해:
- 어떤 순서로 기능을 요청해야 효율적인지
- 모호한 요구사항을 AI가 어떻게 해석하는지
- 오류 발생 시 어떤 맥락 정보를 제공해야 하는지

를 미리 파악하고, **AI와의 대화 방식 자체를 최적화**했습니다.

---

### ③ 프롬프트 블루프린터 (Blueprint) — 기초 구조 설계

코딩을 시작하기 전에 AI와의 대화로 **서비스의 전체 청사진(Blueprint)** 을 먼저 작성했습니다.

```
[Blueprint 작성 과정]

Step 1. 서비스 목적 정의
        → "음악과 영화를 장르 기반으로 개인화 추천하는 웹 서비스"

Step 2. 필요 기능 목록화
        → 회원가입, 로그인, 추천 홈, 둘러보기, 마이페이지, 상세 모달...

Step 3. 기술 스택 결정
        → "백앤드 없이 빠르게 만들 수 있는 스택 추천해줘" → Supabase 선택

Step 4. 데이터 구조 설계
        → 영화 30종 + 음악 30종 데이터셋, 장르 매핑 방식

Step 5. 순차 개발 계획
        → Auth → DB → UI → 이미지 → 폴리싱 순서 확정
```

### Phase 4 — 장르 연동 + 실제 이미지 교체 (핵심 요구사항)

**프롬프트 #5**
```
그 회원가입할 때 장르마다 다양한 음악과 영화들이 나올 거고
그럼 회원가입할 때마다 카드 이미지가 바꾵야 하고
그리고 마이페이지에서 장르 수정하는 경우에도 이미지가 바꾵야 하니까
이거에 맞게 해결해주고
이미지는 AI 사진이 아니라 구글에 관련 이미지들 많이 나올 건데 그걸 사용해줘
```

→ 결과:
- 장르 선택 시 해당 장르 영화/음악 필터링 → 카드 자동 갱신 로직 구현
- IMP Awards · Daum 영화 CDN · Deezer API 활용한 공식 포스터/앉범 칋버 100% 매핑
- PowerShell 다운로드 스크립트(download_assets.ps1) 작성

### Phase 6 — 둘러보기 탭 이미지 오류 수정

**프롬프트 #7**

→ 결과:
- 원인 분석: 추천 홈은 선호 장르 서브셋만 보여주고, 둘러보기는 전체 30곡 표시

- webImage URL 전체 Deezer CDN 주소로 교체

## 4. 느낌점

### 프롬프트 품질 = 결과물 품질

처음에는 "코드 좀 만들어줘" 식의 막연한 요청을 했는데,
목적과 맥락을 충분히 담은 문장으로 바꾸자
AI의 결과물 품질이 눈에 띄게 달라졌습니다.

---

### AI는 "실행자"가 아니라 "협업자"

AI에게 단순히 코드를 시키는 것과,
"왜 이걸 만들어야 하는지"를 함께 이해시키는 것은 완전히 다른 결과를 낙았습니다.

카파시의 말처럼 목표와 제약조건을 주고, AI가 스스로 최선의 방법을 찾게 하는 것이
가장 효율적인 협업 방식임을 실감했습니다.

---

### 프롬프트 블루프린트의 중요성

기능 하나하나를 즉흥적으로 요청할 때와,
전체 서비스의 청사진을 먼저 설계하고 순서대로 요청할 때의 차이는 매우 컸니다.

사전 설계 없이 시작하면 나중에 구조 충돌이 생기고, 수정 비용이 급격히 커집니다.

---

### AI와의 대화는 "질문의 기술"

```
모호한 질문      → 모호한 결과
구체적 질문      → 구체적 결과
맥락 있는 질문   → 창의적이고 완성도 높은 결과
```

단순히 "해줘"가 아니라,
어떤 사용자가 어떤 상황에서 어떤 경험을 원하는지를 담을 때
AI는 단순한 코드 생성기를 넘어 진짜 개발 파트너가 됩니다.

---

### Vibe Coding의 가능성과 한계

하루 만에 인증 + DB + 개인화 추천 + UI를 갖춘 서비스를 만들 수 있었습니다.
이건 분명히 놀라운 생산성입니다.

하지만 동시에:
- AI가 생성한 코드를 이해하지 못하면 오류를 수정하기 어렵습니다
- 프롬프트 실력이 곧 개발 실력이 되는 시대임을 느껼습니다
- 결국 기초 지식 없이는 AI도 제대로 활용하기 어렵습니다
